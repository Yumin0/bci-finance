---

**【背景說明：目前的系統邏輯】**

目前系統裡，「職稱」這件事其實走兩條完全分開的路，這是混亂的根源：

**第一條路（目前管功能權限用的）：**
```
帳號 → 被指派到某個「節點職位」（例如：第三課的課長這個位置）
     → 這個位置對應到「職稱」（課長）
     → 職稱再對應到「系統角色」（課級主管）
     → 系統角色決定這個人能看哪些功能頁面
```

**第二條路（組織架構頁那些綠色名字標籤）：**
```
組織節點 → 有哪些「負責人」（就是節點上那些綠色名字標籤）
         → 但這些負責人「沒有職稱欄位」，跟上面第一條路完全獨立
```

**審核流程現在的問題：**
- 審核步驟設定了「第一步由課長審」，但系統從來沒有真正去驗證「現在登入的人是不是課長」
- 任何人進到審核頁面都可以按同意/退回
- 審核紀錄裡記錄的「審核人」是一個假帳號，不是真實的人
- 沒有「範圍限制」：應該只有四處的課長能審四處的申請，但系統完全沒有做這個判斷

**申請單職稱欄位現在的問題：**
- 職稱下拉清單是從舊的「節點職位組合」撈出來，和現在的組織架構頁是脫節的
- 使用者選到的職稱不一定反映他在組織架構裡的真實職位

---

**【目標：改完後的邏輯】**

```
帳號
 ├─ 在哪個組織節點擔任負責人 + 這個節點的職稱是什麼（例如：四處，處長）
 │    → 審核流程用這條：四處的申請單，往上找到四處的處長來審
 │    → 申請單職稱欄位用這條：自動顯示我在這個節點的職稱
 │
 └─ 系統角色（諮詢議會成員、財務長等特殊角色）
      → 不在組織圖裡，直接指派給帳號
      → 審核流程用這條：需要諮詢議會審的步驟，找所有有這個角色的人
```

---

**【五個修改階段】**

**階段一：職稱管理解除層級限制**
- 現在的問題：職稱（課長、處長等）都被標記了「適用層級」（科/課/處），新增節點職稱時只能選符合層級的職稱。但組織架構的層級已改成自由文字，導致很多節點找不到可用職稱
- 改完之後：職稱就是一個單純的名稱清單，不綁定任何層級，哪個節點都可以選任何職稱
- 資料庫：`role_types` 表的 `level` 欄位停止使用（欄位留著，但 UI 完全忽略它）
- 影響範圍：組織架構頁的「職稱管理」區塊

**階段二：每位負責人直接帶職稱**
- 現在的問題：組織節點上的綠色名字標籤（負責人）沒有職稱欄位，只有一個名字
- 改完之後：每位負責人可以設定他在這個節點的職稱（例如：Chloe → 處長），節點上顯示「處長 Chloe」
- 資料庫：`org_unit_members` 表新增 `role_type_id` 欄位（可空，有些負責人可能沒設職稱）
- 影響範圍：組織架構頁的節點顯示、新增/編輯負責人的操作介面

**階段三：申請單職稱欄位改用新來源**
- 現在的問題：申請單的「職稱」下拉選單是從舊的節點職位組合撈的，跟組織架構脫節
- 改完之後：職稱欄位改為顯示「目前登入的人在組織架構裡有哪些職稱」；若同時在多個節點有不同職稱，讓使用者自己選
- 影響範圍：資金分配申請表單（新增頁、編輯頁）

**階段四：審核流程真正運作**
- 現在的問題①：審核頁面用假帳號記錄「誰審了」，所有審核記錄都掛在同一個假人名下
- 改完之後①：換成真實登入帳號（`session.userId`）
- 現在的問題②：任何人都能點審核按鈕，沒有職稱驗證
- 改完之後②：「組織職位」類型的步驟（如課長審），系統要驗證「目前登入的人在組織架構裡是否擔任課長」才能審；「系統角色」類型的步驟（如諮詢議會成員），驗證帳號是否有這個系統角色
- 現在的問題③：沒有範圍限制，全公司所有課長都看得到所有申請
- 改完之後③：四處的申請，只往四處這條線往上找對應的審核人（課長找同課的課長，處長找同處的處長）；審核管理頁也只顯示符合範圍的申請單
- 影響範圍：三個模組的審核頁（資金分配、付款憑單、暫付款沖銷）、審核管理列表頁

**階段五：側邊欄功能權限查詢更新**
- 現在的問題：系統判斷「我能看哪些功能頁面」走的是舊路徑（職位指派 → 節點職位組合 → 職稱 → 系統角色）
- 改完之後：改走新路徑（組織架構負責人清單 → 直接帶職稱 → 系統角色）
- 影響範圍：`src/app/actions/sidebar-config.ts` 的查詢邏輯

**改動順序：階段一 → 二 → 三 → 五 → 四（審核最複雜，放最後）**
