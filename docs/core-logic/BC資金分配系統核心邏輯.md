## 一、兩張單據的關係（與筑今相同）

```
資金分配申請單（母單，一張）
      │  核准後，可分批開出多張付款憑單，逐筆把額度用掉
      ▼
付款憑單（子單，可多張）
```

- 母單核准後得到一個「核准金額」＝可用額度。
- 底下的付款憑單一張一張把額度用掉，母單的「剩餘金額」隨之減少。
- 剩餘金額用完時，母單自動結案（轉「已付款」）。

## 二、單號規則（沿用筑今基準）

**送單日固定每週三**，方便財務辨識是哪一週的單。

- 資金分配申請日期**只能選週三**，限未來三週內的某一個週三；系統依今天預設最近的送單週三（例：今天 7/13 週一 → 預設 7/15）。
- **資金分配單號 ＝ 申請日期 8 碼 ＋ 3 碼流水序**，例：`20260715001`、`20260715002`。（新系統實際單號即此格式，已對齊。）
- **付款憑單採購單號 ＝ 母單號 11 碼 ＋ 3 碼流水序**，例：母單 `20260715001` → `20260715001001`、`20260715001002`。

## 三、資金分配申請列表

> 這張列表出現在三個地方，欄位完全一致：首頁「我的申請紀錄」、「我的資金分配申請」、「全部申請紀錄」。使用者可用右上角「欄位」按鈕自行開關要看的欄位（會記住個人偏好）。「狀態」「單號」永遠顯示。

| 欄位 | 這欄在講什麼 | 邏輯 / 跨頁關聯 / 用途 |
|---|---|---|
| **狀態** | 這張單走到哪一關：草稿、審核中（會一併顯示目前卡在哪個審核步驟名稱）、已核准、不核准、已付款。 | 顏色與名稱可由「狀態標籤設定」自訂。 |
| **單號** | 這張申請單的正式編號。送出後才有；還是草稿時，這格顯示「繼續編輯」讓人點回去改。 | 全系統辨識這張單、付款憑單掛靠的依據。 |
| **申請處別** | 申請人所屬的「處」。 | 決定要送哪個處的主管審。新增申請時會依申請人所屬單位自動帶入。 |
| **申請課別** | 申請人所屬的「課」。 | 決定課級審核由誰簽。 |
| **申請人** | 送單的人。 | 追責與通知。 |
| **職務** | 顯示「單位＋職稱」，例：新技術研發部 D7-21B 課長。 | 表明申請人身分。 |
| **申請金額** | 申請人一開始填的金額。 | 送審的原始請款數字。 |
| **核准金額** | 審核後真正核准的金額，可能被砍。還沒核准顯示「-」。 | **是後續付款憑單的可用額度上限**。多關審核時，下一關會自動帶入上一關核准的金額當預設（例：課長核 9000，處長預設就帶 9000）；審核者**只能等於或下修，不能往上調高**。 |
| **剩餘金額** | 這張母單還剩多少額度沒用掉。即時算出來的，不是固定存的數字。還沒核准顯示「-」。 | 算法見下方「剩餘金額怎麼算」。歸零時母單自動結案。 |
| **出款帳戶** | 這筆錢從哪個帳戶付。 | 會帶到付款憑單。 |
| **費用項目** | 這筆花費的費用科目（例：2.1 行銷宣傳費用）。 | 會帶到付款憑單，供歸帳。 |
| **項目** | 這筆申請的白話說明（例：剩餘金額邏輯測試）。 | 讓人看懂用途；會帶到付款憑單。 |

### 剩餘金額怎麼算（新系統目前的實際邏輯）

```
剩餘金額 ＝ 核准金額 − 底下每一張付款憑單「目前佔用的金額」加總
```

每張付款憑單「佔用多少」，看它自己走到哪一關：

| 付款憑單狀態 | 佔用金額算法 | 為什麼 |
|---|---|---|
| 草稿 / 審核中 | 用「這張憑單填的金額」 | 一開單就先把額度預扣起來，避免同一筆錢被重複開單用掉。 |
| 已核准 / 已付款 | 用「這張憑單的核准金額」 | 核准階段金額可能被調整，改用核准後的數字。 |
| 不核准 | 算 0 | 這張被退掉，佔用的額度全數退回母單。 |

- 母單還沒核准（沒有核准金額）時，剩餘金額顯示「-」。
- 當財務對某張付款憑單按「確認付款」、且該母單剩餘金額歸零時，母單自動轉為「已付款」結案。剩餘沒用完就一直停在「已核准」，不會提早結案。

> 此算法與筑今基準一致，完整規則見同資料夾《資金分配_剩餘金額計算邏輯規格》。

### 付款憑單的金額是怎麼組成的（重要）

- 付款憑單申請者填的數字欄位有三個：**未稅金額、稅額、總額**。
- 這張憑單的「金額」就是**總額**（一張憑單分多組明細時，是各組總額加總）。
- **審核者審這張憑單時，只針對「總額」按「核准／不核准」，不會、也不能在審核時填或改金額**。這點和資金分配申請單不同（資金分配審核者可以下修核准金額）。

### 這張單其實已經有、但列表沒顯示的資料

- **採購單號**：憑單送出後有正式編號，只是這個精簡列表沒列出來。
- **付款對象（受款人）**：憑單裡有記錄付給誰（可搜尋全部付款對象），列表未顯示。
- **核准金額**：＝這張憑單的「總額」經審核核准後成立的數字（審核只做核准／不核准，不編輯金額）。列表未顯示。

## 七、名詞小抄（避免混淆）

- **母單 / 資金分配申請單**：先講要花多少錢、核准後給一個可用額度。
- **子單 / 付款憑單**：實際付錢給某對象時開的單，一張母單可開多張。
- **申請金額**：一開始填的。**核准金額**：審核後真正准的。**剩餘金額**：母單還沒被用掉的額度（即時算）。
- **佔用**：付款憑單一開出來，就先把母單的額度扣住（預扣），避免重複使用。

## 八、防呆與限制盤點（2026-07-14 全面體檢）

> **這一節是什麼**：把系統裡「跟金額有關的規則」全部攤開來看一次——哪些已經有擋、哪些其實沒擋到、哪些要 Yumin 拍板。
>
> **先講一個重要觀念：「畫面上擋」和「存檔時擋」是兩道不同的門。**
> - 「畫面上擋」＝使用者在填表時，畫面立刻跳提醒、不給按送出。體驗好，但**這道門是可以被繞過的**（例如畫面程式有小錯、或未來改版時不小心弄壞了提醒）。
> - 「存檔時擋」＝不管畫面怎麼放行，系統在真正把資料記下來的那一刻再檢查一次，不合規則就拒收。這才是**保證帳永遠算得對的最後防線**。
> - 理想狀態是兩道門都有。下面盤點會標明每條規則目前是「兩道門都有」還是「只有畫面那道門」。

### 8-1 已經有的防呆（確認沒問題）

| 規則 | 目前狀況 |
|---|---|
| 付款明細每一組的金額必須大於 0 才能送出（資金分配看「費用」、付款憑單看「總額」） | ✅ 有擋。但**存草稿時不檢查**，只有按「送出」才檢查（這是刻意設計：草稿允許先存一半） |
| 同一張母單底下，所有付款憑單的金額加總不能超過核准金額 | ✅ 有擋，而且是「存檔時擋」——建立憑單、修改草稿憑單時系統都會重新算一次剩餘額度，超過就拒收 |
| 財務「確認付款」只能對「已核准」的憑單按，不能重複按、不能跳關 | ✅ 有擋（存檔時擋） |
| 資金分配申請日期只能選週三、限未來三週內 | ✅ 有擋 |
| 暫付款沖銷憑單只能從「已付款＋預支」的付款憑單建立 | ✅ 有擋（畫面與存檔兩道門都有） |

### 8-2 缺口清單與 Yumin 的決定（2026-07-14 已拍板）

> Yumin 拍板的大原則：**職員一律從畫面操作，但存檔時也要再檢查一次，作為最後一道防線**（兩道門都要有）。另外，光是擋下來還不夠——**每次擋下時都要用白話告訴使用者「發生了什麼事、接下來該怎麼做」**，詳見 8-3 提示訊息原則。

#### 🔴 缺口 1：審核時填的「核准金額」，存檔那一刻完全不檢查

文件前面寫的三條規則——「核准金額只能等於或下修、不能往上調高」「付款憑單核准金額不能超過母單剩餘額度」——目前**只有畫面那道門在擋，存檔時完全不看**。

白話說：正常操作下審核人畫面上調不高，但只要畫面程式哪天出一點小錯，或有心人繞過畫面直接送資料，就能核准出「比申請金額還高」「讓母單剩餘變成負數」甚至「0 元、負數」的核准金額，而且系統會照單全收、帳從此算不對。

- [x] **已拍板：要補**。存檔時再檢查一次同樣的規則，不合就拒收。

#### 🔴 缺口 2：付款憑單的金額可以填 0 或負數存進系統

目前「不能超過剩餘額度」有擋，但**沒有擋下限**——金額填 0 甚至負數，存草稿時系統照收（按「送出」才會被「總額必須大於 0」擋住）。

最危險的是**負數**：一張金額 −500 的草稿憑單，會讓母單的剩餘金額**不減反多 500**（因為剩餘＝核准金額−佔用加總，負的佔用等於倒貼回去），全站看到的剩餘金額直接失真。

- [x] **已拍板：要補**。存檔那一刻就規定金額必須大於 0，草稿也不例外。

#### 🔴 缺口 3：母單核准金額被「下修」時，沒檢查是不是已經低於底下憑單用掉的錢

情境：母單核准 5,000 → 承辦人建了一張 3,000 的憑單（審核中，額度已預扣）→ 後面某關審核人把母單核准金額改成 2,000。

此時剩餘金額＝2,000 − 3,000 ＝ **負 1,000**，系統不會攔、也不會提醒任何人。

- [x] **已拍板：擋下並具體說明**。下修核准金額若低於底下憑單已佔用的總額，直接擋下，且提示必須把來龍去脈講清楚——審核人當下很可能根本不記得（或不知道）這張單底下已經有人建過憑單。訊息範例見 8-3。

#### 🟡 缺口 4：暫付款沖銷憑單的金額完全沒有檢查

建立沖銷憑單時填的金額，系統**什麼都不檢查**——可以填 0、負數，甚至填一個比原本預支金額還大的數字。

- [x] **已拍板：要補**。沖銷金額必須大於 0，且不能超過原本那張預支付款憑單的金額。
- 📌 **實作時發現的既有問題（2026-07-14，已一併修正）**：沖銷憑單表單上填的金額欄叫「總額」，但系統存檔時找的是另一個名字的欄位，兩邊對不上——導致**過去建立的沖銷憑單，金額根本沒有存進系統**（現有唯一一張的金額就是空的）。已修正為「表單的『總額』欄＝沖銷金額」（與付款憑單「憑單金額＝總額」同一約定），金額會確實存入，上面的兩條檢查也以這個值把關。

#### 🟡 缺口 5：核准金額填「0」算什麼意思？

- [x] **已拍板：選 A——禁止填 0**。核准金額必須大於 0；審核人要拒絕這張單，請按「不核准」，不是填 0 元核准。

#### 🟢 缺口 6：付款憑單「總額」和「未稅＋稅額」的關係

- [x] **已拍板：總額不可小於「未稅＋稅額」，直接擋下**（不是只提醒）。剛好等於（最常見的正常填法）與大於都放行；直接填總額、未稅稅額留 0 的用法（代墊匯還、國稅局稅額）不受影響。

#### 🟢 缺口 7：兩個人「同一秒」對同一張母單建憑單，理論上可以雙雙過關而超額

剩餘額度檢查是「先看剩多少、再存檔」兩個步驟，如果兩張憑單在幾乎同一瞬間送出，有極小的機率兩張都看到「還夠」而雙雙存進去、加總超額。以目前的使用人數與頻率，實際發生機率非常低。

- [x] **已拍板：先記錄、暫不處理**，等使用量變大再回頭補強。

#### 🆕 缺口 8：額度用完的母單，不該還能建立新的付款憑單（Yumin 補充，對齊筑今）

一張資金分配申請單的剩餘金額用完＝憑單都建完、財務也都付完了，這張單就結束了，**不應該再讓人建立新的付款憑單**。筑今的做法是：這個階段直接**不顯示「建立付款憑單」按鈕**。

新系統現況：
- 母單轉「已付款」後，按鈕**本來就會消失**（按鈕只在「已核准」狀態顯示），這部分已對齊筑今。✅
- 但有一個縫隙：**剩餘金額已經被建好的憑單全數佔用（歸零）、只是還有憑單沒走完付款，母單還停在「已核准」**——這時按鈕還在。點進去雖然存檔時會因超額被擋，但讓使用者白跑一趟、看到擋下訊息也會困惑。

- [x] **已拍板：要補**。剩餘金額歸零（或更早已經 ≤ 0）時就不顯示「建立付款憑單」按鈕，不用等到轉「已付款」。存檔時的超額檢查照舊保留，當最後防線。

### 8-3 提示訊息原則（Yumin 指定，所有防呆共同適用）

**每一條防呆擋下操作時，訊息都必須讓一般職員看得懂三件事：①發生了什麼事 ②為什麼被擋 ③接下來該怎麼做。** 不可以只寫「金額不合法」「驗證失敗」這種話。

而且**訊息裡的人、單、日期都要點名到具體**（Yumin 2026-07-14 補充指定）：

- 要講出**幾月幾號、誰、建立了哪張付款憑單（含採購單號）、金額多少**——不可以只籠統地說「已經有人建立了付款憑單」。
- **不可以用「承辦人」這種抽象稱呼**，一般人看不懂那是誰。要直接寫出那個人的名字（或可操作的組織角色名）；如果建單的人就是看訊息的人自己，就直接寫「你自己建立的」。
- 「接下來該怎麼做」也要指名道姓：直接請那個人去刪除該憑單、或調整該憑單的金額，而不是叫使用者自己去猜要找誰。

以缺口 3 為例（審核人最容易忘記、甚至根本不知道發生什麼事的情境），訊息應該長這樣：

> 「這張資金分配單底下已經有付款憑單在用額度：
> ・7 月 10 日 王小明 建立的「Fireflies 6月平台費」付款憑單（採購單號 20260715001001），金額 3,000 元
> 合計已佔用 3,000 元。核准金額如果改成 2,000 元，會低於已經用掉的錢，帳會對不起來，所以無法儲存。
> 請填 3,000 元以上；或請 王小明 先刪除／調降上面那張憑單的金額，再回來下修。」

若底下有多張憑單，就逐張列出（一張一行）。若建單的人就是目前操作的審核人本人，該行寫「你自己建立的」。

其他缺口的訊息比照同樣的具體程度辦理（擋 0 元時要說「請按不核准，不要填 0 元核准」；擋總額時要說「總額不可低於未稅＋稅額的合計 X 元」……以此類推），實作時逐條寫好給 Yumin 過目。

**呈現方式（Yumin 2026-07-14 拍板）：畫面中央彈窗（全站共用元件）**。理由：長表單的送出按鈕多在頁面底部，錯誤若只顯示在頁頂紅字，使用者按了送出會以為沒反應、要捲回頂部才發現——看不到等於沒做。彈窗不管捲到哪都看得到、按「我知道了」才關閉、裝得下多行點名式長訊息；審核列表原本用瀏覽器原生跳窗（alert）的地方也一併換成同一個彈窗，全站一致。
