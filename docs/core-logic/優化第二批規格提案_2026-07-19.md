## 一、逐項核准金額（Q1，含 Eric 的金額同步回饋）

### 問題本質
資金分配單有多組付款明細，但核准金額只有「整張單一個數字」。核准金額被下修時，系統不知道是哪一組被砍，導致：
1. 轉付款憑單時無法正確帶入各組的費用/稅額（Eric 回饋的根源）
2. 財務看單時對不出下修的是哪個項目

### 提案：審核進度的核准金額改為「逐組核准」
審核頁「審核進度」區塊（現在的單一核准金額框）改為可展開的**逐項核准表格**：

```
審核進度
1. 課長審核  ○不核准 ●核准   [評論]

   逐項核准明細（預設承接上一關，第一關帶申請值）
   ┌────────────────┬──────────┬──────────┬──────────┐
   │ 費用項目（細項）      │ 摘要      │ 申請費用   │ 核准費用（可改）│
   ├────────────────┼──────────┼──────────┼──────────┤
   │ 6.2 數位平台費用     │ Test 01  │ 15,000   │ [15000]  │
   │ 6.2 數位平台費用     │ Test 02  │ 20,000   │ [20000]  │
   └────────────────┴──────────┴──────────┴──────────┘
   稅額依各組稅率設定自動重算，核准金額（總）＝Σ(核准費用＋重算稅額)，不再手填
                                        核准金額：43,750  [確定送出]
```

- 每一關的逐項核准值承接上一關（第一關帶申請單原值），審核人只改要砍的那幾組
- **總核准金額改為自動加總、不可直接手改**（避免總額與逐項對不起來）
- 申請人原始申請值不動（保留在付款明細），核准值另存
- 快速核准／批次核准＝「全部照上一關的逐項值通過」，行為不變

### 資料儲存
- `approval_records.approved_items jsonb`：該步驟各組核准值 `[{ 組序, 核准費用, 重算稅額 }]`（`approved_amount` 照舊存總額，相容既有畫面）
- 轉付款憑單帶入時：讀最新一關的 `approved_items`，逐組帶入未稅金額/稅額、總額＝該組核准值（多組全帶、可刪改；超過剩餘額度照擋）→ **這一步同時解掉 Eric 的金額同步問題**
- 舊單無逐項資料：照舊只帶總額邏輯，不回填

### 需要拍板
- ❓ 下修時「稅額」跟著核准費用重算（依該組稅額選擇），對嗎？
- ❓ 逐項核准表格每一關都顯示，還是預設收合、點開才展開（單子組數多時較清爽）？
- ❓ 與 Riku 名下的「資金轉建立付款單費用帶入」重疊，由誰做要協調

## 二、一次建立多筆付款憑單（Q3）

### 需求兩端
- 請款同事：一張畫面填完所有付款對象，不要一張一張建
- 財務：每個付款對象一個獨立單號才好追蹤

### 提案：批次建立精靈（一張畫面 → 自動拆成 N 張憑單）
資金分配單「建立付款憑單」旁加「**建立多筆付款憑單**」：

1. 進入批次畫面：上方母單對照卡片照舊，下方一個表格，每列＝一個付款對象：
   `受款人｜付款方式｜金額｜摘要｜預支勾選`，＋新增列，底部即時顯示「合計／母單剩餘額度」
2. 按「確定送出」：系統自動產生 N 張**獨立的付款憑單**（各自採購單號流水、各自附件、各走審核流程並同時送審）
3. 審核端不用改架構——審核管理列表已有批次核准，審核人一次勾選核准即可
4. 財務端每張憑單照舊有獨立單號、獨立確認付款

拆單後如需補資料（受款帳號等），各憑單仍可在草稿/審核編輯階段個別修改。

### 需要拍板
- ❓ 這樣「請款端一張畫面、財務端 N 個單號」兩邊都滿足，方向 OK 嗎？（要跟財務確認）
- ❓ 批次畫面上每列要填的欄位清單（上面是我的猜測，以受款人/付款方式/金額/摘要為最小集）
- ❓ 表上寫「Riku 有想法」——先跟 Riku 對過再定案

## 三、付款憑單／沖銷審核可編輯 ＋ 退回重送（Q4）

### (a) 審核頁可編輯（已拍板要做）
把資金分配審核頁的能力搬到付款憑單、暫付款沖銷審核頁：審核人可直接改單子內容、每次儲存記欄位級變更歷程（含附件）、儲存後畫面即時刷新。工程量不小（兩張單的審核頁都要嵌入可編輯表單），預計拆兩批：先付款憑單、再沖銷。

### (b) 退回重送（新流程，需確認細節）
區分兩種否決：
- **不核准**＝這筆費用不符公司條件，單子終結（現況不變）
- **退回**（新增）＝資料不足／當週額度不夠，退給申請人修正後重送，不用重建單

提案：
- 審核動作區多一顆「退回」；新狀態 `returned`（狀態標籤設定可自訂名稱顏色）
- 退回的單：申請人可編輯（含改申請日期到下週）、按「重新送出」→ **從第 1 關重走流程**（中途各關資料已變，讓每一關重看比較安全）
- 審核紀錄保留退回歷程（誰退的、原因），申請人視角審核進度顯示退回原因

### 需要拍板
- ❓ 重送後從第 1 關重走，還是直接回到退回那一關？（我建議重走，簡單且各關都看過最新版）
- ❓ 「退回」資金分配、付款憑單、沖銷三種單都要，還是先做資金分配？
- ❓ 額度不夠退到下週的情境：申請日期由申請人自己改成下週即可，不用系統自動改，對嗎？

## 四、列印／匯出付款憑單（Q5）

### 提案（比照筑今，補財務代印）
- 付款憑單**已付款**後，詳細頁右上角出現「匯出付款憑單」→ 開新分頁 A4 直式 HTML（比照筑今版式：公司抬頭／付款憑單(類型)／申請日期、採購單號／受款人／摘要+會計科目+金額表格／合計／轉帳勾選／受款行+帳戶／CFO·會計·部門主管·申請人簽核欄（帶審核紀錄人名）／單據黏貼處），頁面有「列印」按鈕（瀏覽器列印→可存 PDF）
- **財務付款憑單管理頁（/finance/payment）每列也放「匯出」**——解掉筑今「財務不能代職員輸出」的問題，電子單據財務可直接代印
- 職員輸出後照舊傳回該憑單的「上傳單據」附件欄位

### 需要拍板
- ❓ 版式欄位以筑今截圖為準，但新系統沒有「會計科目」欄位——這欄印空白、拿掉、還是印「費用項目（細項）」？
- ❓ 簽核欄位名（CFO/會計/部門主管/申請人）要照筑今，還是照新系統實際審核流程的關卡名？（我建議照實際關卡）
- ❓ 「已核准但尚未付款」要不要也能匯出（例如先印出來跑紙本流程）？筑今是已付款才有

## 建議的執行順序

1. **列印匯出**（第四節）：獨立、不動資料庫、財務急用度高
2. **逐項核准金額**（第一節）：解掉金額下修對不上與轉憑單帶入兩個痛點
3. **審核可編輯**（第三節 a）：付款憑單 → 沖銷
4. **退回重送**（第三節 b）：動到狀態機，等前面穩定
5. **一次建多筆憑單**（第二節）：等逐項核准金額上線後做，帶入值才正確
