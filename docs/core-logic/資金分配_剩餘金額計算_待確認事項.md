## 一、業務邏輯總結（請先確認這樣理解對不對）

以 Yumin 舉的「Fireflies.ai 6月平台費」例子：

1. **資金分配申請單**：一筆總金額，內含所有相關費用（例如 $2,265 訂閱費 + $33 服務費 + $566 稅額 = $2,864），走完主管到財務長的審核鏈後核准。
2. **核准之後，可以建立好幾張「付款憑單」，各自對應不同的付款對象、各自填不同的金額**：這個例子建了兩張——一張要匯給申請人本人（他先代墊了 $2,298）、一張要付給國稅局（稅額 $566），兩張金額加起來剛好等於資金分配核准的 $2,864。**這些憑單不需要一次全部建完**：也可以是核准後這個月先建一張，下個月有新的支出再建下一張，額度只要還沒用完，隨時都可以再建立新憑單來扣款（例如一筆 5 萬元的出差交際費核准後，分好幾次、每次實際支出時各建一張憑單，直到扣完 5 萬為止；額度用完後如果還要用，就是重新申請一筆全新的資金分配單，不是繼續加掛在原本這張上面）。
3. **每一張付款憑單的金額，是申請人在「建立付款憑單」當下自己算好、自己填的**（他要清楚知道這個對象該拿多少錢，並附上單據/代墊證明），接著這張憑單走主管、財務審核，審核人可以在審核過程調整/確認「核准金額」。**核准之後，財務只負責核對單據、照核准金額全額撥款，不會有財務自行決定只付一部分的情況**——所以「核准金額」跟「實際撥款金額」理論上永遠是同一個數字，不是財務另外決定的。
4. **資金分配申請單的「剩餘金額」＝核准金額 − 目前所有付款憑單已經佔用掉的金額**，會隨著每張憑單的狀態變化（新建、核准、付款、不核准）即時重新計算。
5. **剩餘金額真的變成 0 的那一刻，資金分配單狀態自動變成「已付款」**（結案）。在這之前，不管已經建立、付款過幾張憑單，這張資金分配單都還是「已核准」狀態，可以隨時再建立新的付款憑單繼續扣款，不會提早結案。如果核准金額最後用不完，這張單就會一直停留在「已核准」，不會有「已結案但還有剩餘金額」這種情況。

## 二、新系統現況比對：發現 4 個缺口

我請人去查了新系統（allocation.boptaipei.com）「建立付款憑單」這個功能實際的樣子，跟上面的邏輯對比，落差比我原本以為的更根本：

### 缺口 A：「建立付款憑單」頁面明細欄位填的金額，送出後沒有真的被存進去（真正的問題）
畫面位置：資金分配申請單核准後，點「建立付款憑單」進去的新增頁面，中間的「付款明細」表格（未稅金額／稅額／總額欄位）。
現況：這幾個明細欄位**可以填寫**，畫面上方也會即時顯示費用/手續費/稅額/總額的加總（這部分是對的、跟舊系統一樣）。但按「送出」建立憑單時，程式碼會把這張憑單的「總金額」欄位**直接寫死等於資金分配單的全額**，完全不管你剛剛在明細表格填了什麼數字、加總出多少——也就是說，明細表格填的東西目前只是「畫面上算好看」，並沒有真的存進這張憑單的金額紀錄。
落差：如果你想拆成 $2,298 跟 $566 兩張憑單，就算你把兩張的明細分別填成這兩個數字，資料庫裡兩張憑單存的「總金額」實際上都還是 $2,864（全額）。這就是「同一筆錢被兩張憑單各自重複佔用一次」的真正原因，也是「剩餘金額」沒辦法正確算出來的根本問題。

> **補充修正（Yumin）**：「總額」欄位目前是系統鎖住的唯讀欄位，強制用「未稅金額＋稅額」自動加總算出來，不能手動改。但實際使用情境是：像申請人自己代墊、要匯還給自己的這種對象，會直接填總額（例如 2,298），不會特別去拆未稅金額跟稅額；付給國稅局的稅額也是直接填總額（例如 566）。所以「總額」欄位也要能讓使用者直接手動輸入，不強制用未稅金額＋稅額加總鎖死；但「未稅金額」「稅額」這兩個欄位要保留，因為不確定其他同仁會不會需要拆開填寫的用法。

### 缺口 B：同一張資金分配單可以重複建立付款憑單，但沒有防呆機制
現況：系統目前**沒有擋你**對同一張資金分配單按第二次「建立付款憑單」。但因為缺口 A 的關係，每張憑單存進資料庫的「總金額」都固定是全額，不是使用者實際填的分拆金額。也就是說，就算你手動建立兩張、明細也認真填了 $2,298 跟 $566，資料庫裡兩張憑單的金額紀錄都還是 $2,864，等於同一筆錢被重複佔用兩次。

### 缺口 C：付款憑單審核頁不能調整/確認核准金額
（修正上一版的說法）：付款憑單建立時金額由申請人填好送審（見缺口 A），但目前審核頁按核准/不核准時，畫面上沒有金額輸入框，審核人沒辦法在審核當下調整或確認核准金額（資金分配申請單審核頁就有這個輸入框，可以比照辦理）。
「確認付款」按鈕這一關**不需要**額外的金額輸入——因為財務照核准金額全額撥款，沒有自己決定金額的空間，按鈕維持現狀（直接把狀態改成已付款）即可，不用加彈窗或輸入框。

### 缺口 D：付款憑單頁面沒有「已核准資金分配申請單」對照區塊
你提供的舊系統截圖裡，「建立/編輯付款憑單」跟「審核付款憑單」頁面最上方，都有一塊「已核准資金分配申請單」的明細列（資金分配單號、申請處別/課別、申請人、職稱、核准金額、剩餘金額、幣別、機構、出款帳戶、費用項目、項目），旁邊還有「剩餘金額加總、最終剩餘總額」即時試算，讓職員在建立、審核當下就能對照這張資金分配單「核准了多少、目前還剩多少可以分配」。新系統這兩個頁面**完全沒有**對應的區塊，職員拆分金額時無從對照。

## 三、這代表要新增的功能範圍

這次不是三選一的小問題，而是要新增以下幾塊功能（打勾表示你確認這塊要做）：

- [ ] **「總額」欄位改成可以手動輸入**，不強制鎖住用未稅金額＋稅額自動加總（未稅金額、稅額欄位保留，供需要拆開填寫的同仁使用，但不強迫）。
- [ ] **修正「建立付款憑單」送出時的存檔邏輯**：改成把使用者在明細表格填的總額，真的存進這張憑單的金額欄位，而不是像現在這樣固定寫死等於資金分配單全額。
- [ ] **建立/新增付款憑單時，限制「這張＋同一張資金分配單底下已經建立的其他憑單」金額加總，不能超過資金分配單的核准金額**，避免同一筆錢被重複分配出去。
- [ ] **付款憑單新增「核准金額」欄位**：審核時可以填/改核准金額（比照資金分配申請單審核頁現有的做法，預設帶入申請人建立時填的金額）。「確認付款」按鈕維持現狀不用改，因為財務只照核准金額撥款，沒有自行決定金額的情況。
- [ ] **「建立付款憑單」「審核付款憑單」頁面補上「已核准資金分配申請單」對照區塊**，顯示這張資金分配單的核准金額、目前還剩多少可以分配，比照你截圖裡舊系統的設計。
- [ ] **資金分配申請單列表上「剩餘金額」這個目前空著的欄位，改成用上面這一整套邏輯即時算出來顯示**。
- [ ] **剩餘金額變成 0 時，資金分配單自動標示為「已付款」**（不是「所有現有憑單都已付款」，因為隨時可能再建立新憑單）——這題延續下面的兩個選項討論。

## 四、資金分配單「已付款(結案)」怎麼表示？

目前資金分配申請單的狀態只有四種：草稿、新單（待審核）、已核准、未核准。付款憑單則已經有「已付款」這第五種狀態（在「狀態標籤設定」頁面可以看到付款憑單模組已經有這一行，資金分配申請模組還沒有）。

- [ ] ~~選項 A：狀態本身不變，另外加一個「已結案」標記~~

- [x] **選項 B（Yumin 已決定）：新增一個真正的狀態「已付款」**
  申請單狀態畫面上會直接顯示「已付款」這個新狀態（跟付款憑單模組一樣，在「狀態標籤設定」頁面也會多出這一行可以自訂名稱顏色），剩餘金額變成 0 時自動從「已核准」轉過去。開發時需要逐一檢查全站約 21 處判斷「這張單是不是已核准」的地方，確認加了這個新狀態後不會顯示錯誤或漏掉。

## 五、附帶提醒（本來就有的已知限制，本次先不用一起決定）

- 如果某張付款憑單被「不核准」，承辦人卻沒有重新建一張新的，這張資金分配申請單就會永遠卡在「已核准」，沒辦法自動變成「已結案」。之後要靠通知功能提醒承辦人處理，這次先不做。
- 如果一開始核准的金額本來就抓得比較高，等全部真的付完款後，「剩餘金額」可能還是大於 0（不是剛好歸零）。這筆多出來的錢，之後要不要有辦法「退回」給某個總的資金池繼續用，還是就讓它一直顯示在這張已結案的單子上、不用做任何動作？也想聽聽你的想法。
