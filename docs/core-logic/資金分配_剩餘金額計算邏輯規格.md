## 0. 核心結論

剩餘金額不是靜態欄位,是**每次相關事件發生時即時重算**出來的衍生值。

```
剩餘金額 = 資金分配核准金額 − Σ(所有「有效佔用中」付款憑單的佔用金額)
```

「佔用金額」的計算基準會隨付款憑單自己的生命週期換算(申請金額 → 核准金額 → 實際付款金額),**不核准**則歸零退回。

> ⚠️ 系統中只有「不核准」這一種被拒絕的狀態,**沒有「駁回」「作廢」等其他名稱或其他獨立狀態**,不要把它們當成不同的狀態各自處理。

## 1. 兩層單據關係

```
資金分配申請單(1)  ──可拆分建立──▶  付款憑單(多筆)
   │                                    │
   ├ 申請金額(職員填)                    ├ 申請金額(職員填)
   ├ 核准金額(財務長核定,可能≠申請金額)  ├ 核准金額(主管/財務核定,可能≠申請金額)
   └ 剩餘金額(動態計算)                  └ 實際付款金額(付款時填,受下方限制)
```

兩層單據都走「職員送出 → 主管審核 → 財務最終核准」審核鏈,但**剩餘金額只掛在資金分配單上**。

---

## 2. 【硬性限制】實際付款金額 ≤ 核准金額

財務人員已確認:**單張付款憑單的實際付款金額,絕對不可能超過該憑單自己的核准金額。**

這是一條**寫入層級的驗證規則(validation)**,必須在付款動作發生時強制檢查,不是靠邏輯運算自然避免:

```
function recordActualPayment(voucher, actual_amount):
    if actual_amount > voucher.approved_amount:
        reject("實際付款金額不可超過核准金額")
    voucher.actual_paid_amount = actual_amount
    voucher.status = 'paid'
    recalculate(voucher.fund_allocation)
    checkAllocationCloseCondition(voucher.fund_allocation)
```

**這條限制的意義**:保證單張憑單付款完成後,佔用金額只會「不變或減少」,不會突然暴增,剩餘金額不會因單張付款而變成負數。

> 注意:這條限制只保證「單張憑單」層級不超付。它不保證「資金分配單」整體最後剛好歸零 —— 如果一開始核准金額本來就估高了,結案時剩餘金額仍然可能大於 0,這是正常情況,見第 4 節。

## 3. 付款憑單狀態 → 佔用金額對照表

| 付款憑單狀態 | 佔用金額用哪個欄位 | 說明 |
|---|---|---|
| 新單 / 審核中 | **申請金額** | 一建立就先預扣,避免同一筆錢被重複申請 |
| 已核准(尚未付款) | **核准金額** | 主管/財務可能調整過金額,調整後取代申請值 |
| 已付款 | **實際付款金額** | 受第 2 節限制,必然 ≤ 核准金額 |
| 不核准 | **0** | 金額全數退回資金分配單的剩餘額度 |

```
function occupiedAmount(voucher):
    if voucher.status == 'rejected':  # 對應「不核准」,系統只有這一種被拒絕的狀態
        return 0
    if voucher.status == 'paid':
        return voucher.actual_paid_amount
    if voucher.status == 'approved':
        return voucher.approved_amount
    # draft / pending_manager / pending_finance
    return voucher.requested_amount

function remainingAmount(allocation):
    if allocation.approved_amount is null:
        return null   # 前端顯示 "-"
    occupied = sum(occupiedAmount(v) for v in allocation.vouchers)
    return allocation.approved_amount - occupied
```

## 4. 資金分配單狀態機(含結案條件)

| 狀態 | 進入條件 | 剩餘金額顯示 |
|---|---|---|
| 新單 / 未核准 | 職員送出,尚未走完審核 or 被不核准 | `-` |
| 已核准-財務長 | 財務長核准,且尚有 ≥1 筆付款憑單未達「已付款」 | 依第 0 節公式即時計算 |
| **已付款(結案)** | **該資金分配單底下「所有」付款憑單狀態皆為 已付款**(不論剩餘金額是否為 0) | 停止變動,固定顯示結算當下的剩餘金額 |

```
function checkAllocationCloseCondition(allocation):
    if allocation.status != '已核准-財務長':
        return
    vouchers = getVouchersByAllocation(allocation.id)
    if len(vouchers) == 0:
        return  # 尚無憑單,不可能結案
    if all(v.status == 'paid' for v in vouchers):
        allocation.status = '已付款'
        allocation.remaining_amount_final = remainingAmount(allocation)
```

**本版結案規則(嚴格版,已確認)**:
- 結案**只看「底下所有付款憑單狀態是否都是已付款」,不要求剩餘金額必須等於 0**。
- 若一開始核准金額估高,全部付款完成後剩餘金額 > 0,屬正常情況,系統**不需要**額外提示或阻擋結案,直接轉為「已付款」狀態即可。
- 「所有」的判斷**不含例外**:只要有任何一筆憑單狀態是不核准(不是 paid),該資金分配單就**不會**自動結案,會持續停留在「已核准-財務長」。這是刻意保留的已知限制,不在本版邏輯裡自動處理,見第 7 節待辦。

## 5. 事件驅動重算規則

每次以下事件發生,立即重算該資金分配單的剩餘金額,並檢查結案條件:

1. 建立新付款憑單(狀態=新單)→ 扣「申請金額」
2. 付款憑單核准,金額可能被調整(狀態=已核准)→ 改扣「核准金額」
3. 付款憑單被不核准 → 該筆佔用金額歸零,退回剩餘金額
4. 付款憑單完成付款(狀態=已付款)→ 改扣「實際付款金額」(≤核准金額,見第2節限制)→ 觸發結案條件檢查

### 驗算(財務人員案例,資金分配核准金額 $5,000)

| 日期 | 事件 | 剩餘金額 |
|---|---|---|
| 7/2 | 資金分配核准 | $5,000 |
| 7/3 | 建立付款單A,申請$1,000(新單) | $4,000 |
| 7/4 | 建立付款單B,申請$800(新單) | $3,200 |
| 7/5 | 付款單A被不核准 | $4,200 |
| 7/6 | 付款單B核准$800 | $4,200(不變) |
