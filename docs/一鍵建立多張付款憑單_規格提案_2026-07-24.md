# 一鍵建立多張付款憑單 規格提案（2026-07-24）

> 接續 `docs/core-logic/優化第二批規格提案_2026-07-19.md` 第二節（Q3）。
> 本版依 Yumin 提供的筑今系統實際使用截圖（業務處 SBBG 台中場活動請款）重寫，
> 用真實情境把 Before / After 寫清楚，供 Yumin、Riku 與財務拍板。

## 一、真實情境（以 SBBG 台中場為例）

業務處每週辦外部講座。承辦人廖芳欣不想多跑「暫付款沖銷」流程，所以採用這個做法：

1. **活動前**：先申請一張資金分配單，預估整場活動總金額。
   - 單號 20260708002，核准金額 25,800
   - 付款明細是「費用類別」導向：高鐵 10,500／計程車 1,800／顧問餐飲 3,500／Lalamove 運費 9,000／耗材與緊急預備金 1,000
2. **活動中**：組員各自先墊錢（自己買高鐵票、自己付計程車、餐費）。
3. **活動後**：收齊所有人的發票，**按「人」建立付款憑單還錢**——誰墊了錢，就開一張以他為受款人的憑單。
   - 這場活動 6 個人墊錢，所以要建 6 張憑單：
     黎裕喬 1,175／蔡奇軒 2,340／莊苡甄 1,815／林冠成 1,900／鄭雅云 2,000／廖芳欣 11,105
   - 合計 20,335，母單剩餘 5,465（25,800 − 20,335）
4. **每張憑單裡面又有多筆明細**：一人可能墊了好幾筆（例如林冠成：餐費 500＋高鐵 700＋高鐵 700），
   選受款人後自動帶出銀行／分行／帳號／e-mail，發票也是各自上傳到自己那張憑單。
5. **剩餘的 5,465 其實是要付給老闆的**：其中一筆費用的支付對象是老闆，但老闆不會自己整理
   付款明細、收據，一律請財務部的人代為處理。而筑今的規則是**只有申請人本人（廖芳欣）能從
   自己的資金分配單轉付款憑單**，財務不能代建——所以財務只好**另外申請一張 5,465 的資金分配單**、
   重走一次完整審核，再轉憑單付給老闆。同一筆活動預算被拆成兩張母單，純粹是權限限制造成的
   （這個問題的處理方向見「六、延伸議題」第 1 點）。

**關鍵觀察**：資金分配單的明細是「費用類別」、付款憑單的拆分是「付款對象（人）」，
兩邊維度不同、對不起來。拆成幾張憑單、每張多少錢，只有承辦人自己知道。

## 二、Before：現在要怎麼做（筑今與新系統都一樣痛）

筑今的資金分配單底部其實已經有「建立多張付款憑單」按鈕，但它只是**一次產生 N 張空殼**，
之後每張還是要一張一張點進去補資料。我們新系統目前連批次產殼都沒有，只能單張建立。

以 SBBG 台中場 6 張憑單為例，承辦人要重複做 6 輪：

```
第 1 輪：進母單 → 建立付款憑單 → 填明細（餐費 500）→ 選受款人黎裕喬
         → 上傳黎裕喬的發票 → 送出
第 2 輪：進母單 → 建立付款憑單 → 填明細（高鐵 670＋高鐵 670＋餐費 500＋計程車 245＋計程車 255）
         → 選受款人蔡奇軒 → 上傳蔡奇軒的發票 → 送出
第 3 輪：……（同上，換莊苡甄）
第 4~6 輪：……
```

主管端也一樣：6 張憑單要一張一張點進去審核，不能有遺漏。

### 痛點整理

| 角色 | 需求／痛點 |
|---|---|
| 承辦人 | 同一場活動要進出建單頁 6 次，每次都重填一遍摘要／受款人／金額／傳檔 |
| 主管 | 6 張單要逐張點進去核准，漏一張就卡住一個人領不到錢 |
| 財務 | 每人一張獨立單號要保留（方便逐筆追蹤撥款）；且**主管（課級／處級）必須進到每一張憑單的內頁看過明細金額才能核准**——從外層列表直接批次核准等於沒看內容就放行，財務不接受 |

三方需求其實不衝突，這個功能的目標是：
**請款端一張畫面填完 → 系統拆成 N 個獨立單號（財務照舊逐筆追蹤）→ 主管在一頁看完全部內容、逐張核准**。

## 三、After：整體長怎樣（新增三個畫面，原流程原樣不動）

先講大原則：**現有的單張建單頁、單張審核頁、外層列表，邏輯與樣式通通不動**。
本案是針對「一張資金分配單對多張付款憑單」的特殊情境，**額外新增**三個畫面：

| 新增畫面 | 給誰用 | 解什麼問題 |
|---|---|---|
| 3-1 批次建單頁 | 申請人 | 一張畫面填完 6 個人的憑單，一次送出 |
| 3-2 依母單分組視圖 | 申請人 | 列表不用再肉眼比對採購單號前綴歸類 |
| 3-3 合併審核頁 | 主管 | 一頁看完 6 張完整內容、逐張核准 |

沒有「一母單多憑單」需求的人，用起來完全感覺不到系統有變。

### 3-1 建單端：批次建單頁

**入口**：母單（已核准的資金分配單）編輯頁，「建立付款憑單」旁邊多一顆
「**建立多張付款憑單**」。單張建立照舊存在，兩顆按鈕並存。

**用 SBBG 台中場走一遍**：

1. 廖芳欣進母單 20260708002 編輯頁，點「建立多張付款憑單」
2. 進到批次建單頁。主體是一個**付款對象列表**——一列＝一個付款對象＝送出後的一張憑單。
   收合狀態整排只顯示「受款人｜付款方式｜明細筆數｜小計」，6 個人與各自金額**一眼掃完**；
   點「展開」才攤開該人的付款明細表格與附件上傳（用單張建單頁同一套元件編輯）。
   受款人選定後，銀行／分行／帳號／e-mail 依付款對象資料**自動帶入存進憑單，
   但批次頁不顯示**（畫面保持乾淨，要看去單張憑單頁）：

```
┌─ 已核准資金分配申請單（對照卡片，同單張建單頁）──────────────────┐
│ 20260708002｜SBBG 台中場 7/21 活動費用｜核准 25,800｜剩餘 25,800  │
└─────────────────────────────────────────────────────────┘

付款對象列表（一列＝將產生一張付款憑單）

┌───┬────────────────┬──────────┬──────┬──────────┬─────────┬───────────┐
│ # │ 受款人          │ 付款方式  │ 預支 │ 明細筆數 │ 小計    │           │
├───┼────────────────┼──────────┼──────┼──────────┼─────────┼───────────┤
│ 1 │ [黎裕喬_E0010▼] │ [銀行轉帳▼]│ □   │ 2 筆     │ 1,175   │ ▼收合　✕ │
│   │ ├ 明細： 高鐵費用 675／餐費 500 （表格可增刪列，同單張建單頁）        │
│   │ └ 附件： [上傳黎裕喬的發票…]                                       │
│ 2 │ [蔡奇軒▼]       │ [銀行轉帳▼]│ □   │ 5 筆     │ 2,340   │ ▶展開　✕ │
│ 3 │ [莊苡甄▼]       │ [銀行轉帳▼]│ □   │ 2 筆     │ 1,815   │ ▶展開　✕ │
│ 4 │ [林冠成▼]       │ [銀行轉帳▼]│ □   │ 3 筆     │ 1,900   │ ▶展開　✕ │
│ 5 │ [鄭雅云▼]       │ [銀行轉帳▼]│ □   │ 2 筆     │ 2,000   │ ▶展開　✕ │
│ 6 │ [廖芳欣▼]       │ [銀行轉帳▼]│ □   │ 7 筆     │ 11,105  │ ▶展開　✕ │
└───┴────────────────┴──────────┴──────┴──────────┴─────────┴───────────┘
　＋新增付款對象

　全部合計：20,335 元／母單剩餘額度：25,800 元（送出後剩餘：5,465 元）

　[儲存草稿]　[確定送出]
```

   收合列的「小計」＝該人明細各列總額加總（即該張憑單的憑單金額），
   跟底部「全部合計」即時連動——要對「每個人拿多少、加起來對不對」不用展開任何一列。

3. 全部填完按「確定送出」→ 確認彈窗「將建立 6 張付款憑單並送出審核」→ 確認後
   系統一次建立 6 張**各自獨立**的付款憑單、同時送審
4. 回到「我的付款憑單」列表：出現 6 筆新單（20260708002001～006）

**「拆成 N 張獨立單號」是什麼意思**：畫面上你只填了一頁，但按下送出那一刻，
資料庫產生的是 6 張互相獨立的付款憑單——各有自己的採購單號流水（沿用現有
`nextPurchaseOrderNumber`）、自己的明細、自己的附件、自己的審核流程。
**送出之後批次頁就功成身退**，這 6 張跟手動建 6 次的結果沒有任何差別：
要補資料進單張草稿／審核編輯、要退回就退那一張、財務照舊逐張確認付款。
技術上不新增任何資料表、不改審核架構——批次頁只是「一次呼叫 N 次現有的建單邏輯」。

**金額防線**：

- 送出時檢查「N 張合計 ≤ 母單剩餘額度」（沿用 `checkAmountWithinRemaining`，一次驗全部），
  超額擋下並照現有格式點名是哪幾張、超多少
- 每張的總額必須 > 0（沿用 `validateFeePositive`）；儲存草稿放寬（同單張建立頁）
- **逐項核准帶入在批次模式不適用**：母單明細是費用類別、憑單按人拆，對不起來，
  所以批次畫面金額全部由承辦人自己填，不做自動帶入（單張建立的帶入邏輯不動）

### 3-2 申請人列表：依母單分組視圖（解「肉眼比對單號前綴會脫窗」）

**現況問題**：「我的付款憑單」列表一列一張憑單，哪些憑單屬於同一張母單，
只能自己盯著採購單號開頭「20260722002…」歸類，單多的時候眼睛脫窗。

**After**：列表右上加一個檢視切換「**一般列表｜依資金分配單分組**」。
預設仍是一般列表（原樣不動），切到分組視圖長這樣：

```
[一般列表]｜[依資金分配單分組] ← 檢視切換

▼ 母單 20260708002　SBBG 台中場 2026年7月21日活動費用（NTC）
　 核准金額 25,800｜已建憑單 6 張・合計 20,335｜剩餘 5,465
　 ┌ 狀態　　　　　│ 採購單號　　　　│ 付款對象 │ 申請金額 │ 核准金額 ┐
　 │ 新單-課長審核 │ 20260708002001 │ 黎裕喬　 │ 1,175   │ -       │
　 │ 新單-課長審核 │ 20260708002002 │ 蔡奇軒　 │ 2,340   │ -       │
　 │ …（共 6 列，欄位同一般列表，點採購單號進單張詳細頁）　　　　　　 │
　 └──────────────────────────────────────────────────┘
▼ 母單 20260722002　附檔測試
　 核准金額 …｜已建憑單 3 張・合計 …｜剩餘 …
　 │ …
```

- 一張母單一個收合區塊：標題列＝母單號＋項目名＋三個金額（核准／已建合計／剩餘），
  同一場活動的錢花到哪、還剩多少，一眼看完
- 底下憑單列的欄位、連結行為都跟一般列表相同，只是換個分組方式排

### 3-3 審核端：合併審核頁

**現況問題**：主管收到 6 張待審，要「點進第 1 張 → 審 → 回列表 → 點第 2 張 → …」
重複 6 次；而財務要求「必須看過每張內容才能核准」，所以外層快速核准也不能用。

**After 分兩步**。第一步，審核管理清單同母單分組（長相同 3-2），
組標題旁多一顆「**一起審核**」：

```
▼ 母單 20260708002　SBBG 台中場…　待審核 6 張　　　　　[一起審核]
　 │ 新單-課長審核 │ 20260708002001 │ 黎裕喬 │ 1,175 │ …
　 │ …（單張的「審核」入口照舊保留）
```

第二步，點「一起審核」進**合併審核頁**。可以想像成：**把 6 個單張審核頁直向疊在同一頁**：

```
┌ 母單對照卡片：20260708002｜核准 25,800｜本頁 6 張合計 20,335 ┐

━━ 第 1 張／共 6 張：20260708002001（黎裕喬｜1,175）━━━━━━━━
　 付款明細：高鐵費用 675／餐費 500（表格，同單張審核頁）
　 受款人：黎裕喬_E0010｜銀行 822｜分行 0602｜帳號 602540373792
　 附件：發票1.jpg　發票2.jpg（可點開預覽）
　 課長審核：○不核准 ●核准｜評論[＿＿＿]｜核准金額[1,175]｜[確定送出]

━━ 第 2 張／共 6 張：20260708002002（蔡奇軒｜2,340）━━━━━━━━
　 …（同上結構）

　 …

━━ 第 6 張／共 6 張：20260708002005（廖芳欣｜11,105）━━━━━━━
```

- 每一張的內容顯示＝現有單張審核頁**原樣**：明細表格、受款人銀行資訊、附件全部攤開，
  滿足財務「看過內容才核准」的要求
- 每一張的審核列＝現有單張審核頁那一列（核准／不核准、評論、核准金額、確定送出），
  **各自獨立送出、各自寫一筆 `approval_records`**——對系統來說，跟主管進 6 次
  單張審核頁做的事完全等價，差別只是不用進出列表 6 次
- 審完一張，該張區塊反灰標「已核准」，往下捲繼續審下一張；頁面頂端顯示進度「已審 2／6」
- 想單獨細看某張，點採購單號照舊進單張審核頁

### 外層列表的快速核准怎麼辦（唯一會動到現有畫面的點，待財務拍板）

配合財務「必須看內容」的要求，付款憑單審核清單的**快速核准／批次核准按鈕建議移除**
（或至少對課級／處級主管移除），一律進審核頁（單張或合併頁）才能核准。
資金分配申請單的快速核准不在本案範圍、維持現狀。除此之外，本案不改任何現有畫面。
→ 列入待拍板第 3 點。

| 額度控管 | 每張送出時擋超額 | 送出時 6 張合計一起擋超額 |

## 四、需要拍板的事

> 本功能由 **Yumin 負責開發**（2026-07-24 拍板），暫不需與 Riku 核對想法。

1. **方向確認（要問財務）**：
   - 建單端「一張畫面 → 拆成 N 張獨立單號」——財務追蹤方式跟現在完全一樣，OK 嗎？
   - 審核端「合併審核頁」——主管在一頁看到每張憑單的**完整內容**（明細、金額、附件）
     並逐張核准，每張獨立寫審核紀錄。這樣算不算滿足「必須進到每一張看過才核准」？
     （我們的理解：財務在意的是「核准前看過內容」，不是「一定要分 6 次進 6 個網址」）
2. **草稿行為**：批次畫面的「儲存草稿」＝一次建 N 張草稿憑單（之後可逐張補件送出），
   還是整包批次畫面暫存、下次回來繼續編？
   **我建議前者**（直接建 N 張草稿）：不用新資料表，拆出來就是普通草稿，沿用所有現有頁面。
   代價是草稿一旦拆出去，就不能回到批次畫面整包編（要逐張編）。
3. **外層快速核准的存廢**：配合財務要求，付款憑單審核清單的快速核准／批次核准是
   **全部移除**，還是**只對課級／處級主管移除**（群組步驟如支出課／財務人員保留）？
   另外：暫付款沖銷憑單的審核清單要不要比照辦理？
4. **合併審核頁的範圍**：只做「同母單多張 pending 憑單」的情境，還是主管的待審清單
   任意多張都能合併審？**我建議先只做同母單**（這是本案情境，且頁面頂端的
   「合計 vs 母單核准金額」對帳才有意義）。
5. **分組視圖的細節**：
   - 只有一張憑單的母單，要不要也照分組顯示？**我建議照分組**（規則一致，不用解釋例外）
   - 分組視圖做成「我的付款憑單」頁內的檢視切換即可，還是要獨立側邊欄頁面？
     **我建議頁內切換**（少一個選單項目，切換也快）
   - 「全部付款紀錄」與審核管理清單要不要同步提供分組視圖？（審核清單 3-3 已含，

## 六、延伸議題（本案不做，先記錄，列下一批待辦）

### 1. 財務代建付款憑單（解掉「老闆那筆要另立母單」的問題）

**現況（Before）**：只有申請人本人能從自己的資金分配單建立付款憑單。遇到「支付對象是老闆、
資料由財務代整理」的情況，財務動不了廖芳欣的單，只能**另外申請一張 5,465 的資金分配單**
重走完整審核，再轉憑單付老闆。結果：同一場活動的預算拆在兩張母單、多跑一整輪審核，
原單的 5,465 剩餘也永遠用不掉、無法正常結案。

**未來方向（After）**：開放財務（或指定角色）在**任何人的已核准資金分配單**下建立付款憑單
（單張或批次畫面皆可進），額度照樣吃原母單的剩餘——老闆那筆直接從 25,800 裡出，
母單剩餘歸零、自動轉「已付款」結案，不用再立第二張母單。
先例：2026-07-20 已做過「財務代職員匯出列印憑單」，解的是同一類筑今權限限制。

**為什麼先不做**：Yumin 判斷不急。而且要先想清楚兩件事——
代建的憑單「申請人」欄掛誰（掛原申請人？掛財務？影響審核人解析與通知對象）、
以及入口放哪（財務頁？全部申請紀錄？）。等本案上線後拍板。
若拍板時想順路做：搭著批次建單畫面加權限判斷即可，增量成本低。

### 2. 年度餘額大清理（核准未用餘額釋回）

**現況**：每張核准的資金分配單，實際沒用完的剩餘金額會一直掛在單上（狀態停在 approved、
不會結案）。整年累積下來是一筆不小的「帳面已核准、實際沒花掉」的錢，
財務年底要大清理一波，把這些餘額轉回可動用資金。

**未來方向**：提供財務一個「餘額釋回／強制結案」操作（例如年底批次把帶餘額的 approved 單
結案並記錄釋回金額），讓帳面核准額度回歸真實。會動到狀態機與結案規則，需要獨立規格，
明確列下一批。
